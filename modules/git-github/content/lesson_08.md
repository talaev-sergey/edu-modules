## Ход занятия

### 1. Организационный момент (5 мин)

**Действия преподавателя:**

Поприветствовать группу, проверить, что у каждого ученика включён компьютер, установлены Git и VS Code (с урока 1), есть вход в аккаунт на GitHub (с урока 5). Сегодня главная проверка — доступ к общему репозиторию класса: каждый ученик должен быть добавлен в него как участник с правом записи (коллаборатор) и принять приглашение. Напомнить путь модуля простыми словами: «Урок 1 — мы поставили Git. Дальше учились делать репозиторий, коммиты, ветки, выкладывать проект на GitHub, скачивать чужие проекты, работать в команде через Pull Request. Сегодня — финал: все вместе соберём один общий сайт класса и опубликуем его в интернете». Открыть на проекторе общий сайт класса (пока на нём только главная страница со списком класса, личных страниц ещё нет) и общий репозиторий. Назвать цель урока: «На сайте уже есть список всего класса, но личных страниц пока нет. Сегодня каждый создаст и сверстает свою страницу — напишете о себе — и через Pull Request добавите её в общий сайт. В конце урока откроем сайт и увидим страницы всего класса». Показать на проекторе два адреса, они понадобятся весь урок: адрес общего сайта класса и адрес общего репозитория.

> Подготовка до урока (обязательно):
>
> 1. Создать общий репозиторий класса с именем `class-site-<имя группы латиницей>` (например, `class-site-vtornik`) на аккаунте преподавателя и загрузить в него заранее подготовленную папку сайта — главную страницу `index.html` и папку `pages` с файлом-шаблоном `temp.html` (образец личной страницы, ниже). Свои страницы ученики создадут на уроке сами, копируя код из `temp.html`.
> 2. Добавить всех учеников коллабораторами с правом записи: **Settings — Collaborators — Add people** (по имени их аккаунта с урока 5). Попросить учеников заранее принять приглашение (приходит на почту и в уведомления на GitHub) — иначе на уроке они не смогут отправить ветку.
> 3. Проверить, что в `index.html` уже есть список всего класса со ссылками на личные страницы `pages/<фамилия латиницей>.html` (образец ниже). Каждому ученику назначить и заранее сообщить имя его файла — латиницей, маленькими буквами, без пробелов (например, `ivanov.html`): кириллица и пробелы в адресах сайта работают плохо. Имя файла, который создаст ученик, должно точно совпадать со ссылкой в `index.html`.
> 4. Включить GitHub Pages: **Settings — Pages — Source: Deploy from a branch — Branch: main / (root) — Save**. Через минуту сайт станет доступен по адресу вида `https://<аккаунт преподавателя>.github.io/class-site-<имя группы>/`. Этот адрес и адрес репозитория показать на проекторе.
>
> Образец главной страницы `index.html`:
>
> ```html
> <!DOCTYPE html>
> <html lang="ru">
> <head>
>   <meta charset="utf-8">
>   <title>Наш класс</title>
> </head>
> <body>
>   <h1>Наш класс</h1>
>   <p>Странички учеников:</p>
>   <ul>
>     <li><a href="pages/<фамилия ученика>.html">Фамилия Имя ученика</a></li>
>     <!-- строка на каждого ученика -->
>   </ul>
> </body>
> </html>
> ```
>
> Содержимое файла-шаблона `pages/temp.html` — на уроке каждый ученик копирует этот код в свой новый файл `pages/<своя фамилия латиницей>.html` и меняет текст под себя:
>
> ```html
> <!DOCTYPE html>
> <html lang="ru">
> <head>
>   <meta charset="utf-8">
>   <title>Моя страница</title>
> </head>
> <body>
>   <h1>Привет! Меня зовут ___</h1>           <!-- впиши своё имя -->
>   <p>Мне ___ лет, учусь в ___ классе.</p>   <!-- впиши возраст и класс -->
>   <p>Я люблю ___.</p>                        <!-- впиши своё увлечение -->
>   <p><a href="../index.html">На главную</a></p>
> </body>
> </html>
> ```

### 2. Теоретическая часть (10 мин)

> Режим Б: здесь даём только короткое введение — что такое GitHub Pages и как устроен общий сайт класса. Все действия (клонировать, сверстать страницу, отправить через Pull Request) ученики освоят руками в практике по схеме «мини-теория — задание».

**Действия преподавателя:**

Объяснить на экране две идеи: что такое GitHub Pages и как из одного репозитория получается общий сайт класса. Опираться на то, что ученики уже умеют (репозиторий, ветки, Pull Request, merge — уроки 4–7). Подробности разберём по ходу практики.

#### GitHub Pages — публикация сайта прямо из репозитория

Обычный сайт нужно где-то разместить, чтобы он открывался у всех в интернете по адресу. GitHub умеет делать это бесплатно прямо из репозитория: эта возможность называется GitHub Pages. Если в репозитории лежат файлы страниц (формата `.html`), Pages превращает их в настоящий сайт с собственным адресом. Это статический сайт: набор готовых страниц, как книга со страницами, которые просто открываются по очереди.

#### Один репозиторий — общий сайт класса

У нас один общий репозиторий, и все вы — его участники (коллабораторы) с правом записи. Главная страница `index.html` уже готова: на ней список всего класса со ссылками. А вот личных страниц в папке `pages` ещё нет — каждый создаст свою сам. Ваша задача сегодня — сверстать свою страницу (написать о себе) и добавить её в общий сайт. Делать это будем по-командному, как на прошлом уроке: каждый в своей ветке, через Pull Request, который преподаватель проверит и вольёт (merge). Так над общими проектами работают настоящие команды: меняют не сразу общий результат, а через предложение и проверку.

#### Записи в блокнот

**GitHub Pages** — бесплатная публикация сайта прямо из репозитория на GitHub; страницы становятся доступны в интернете по своему адресу.

**Статический сайт** — сайт из готовых файлов-страниц (HTML).

**index.html** — главная страница сайта, которая открывается по основному адресу.

**Коллаборатор** — участник с правом записи в общий репозиторий.

### 3. Практическая работа (60 мин)

**Действия преподавателя:**

Работаем по принципу «Я показываю — делаем вместе — делаешь сам». Каждый новый шаг сначала показать на проекторе, затем ученики повторяют у себя и проверяют результат. Сегодня соединяются почти все навыки модуля: клонирование (урок 6), ветки (урок 4), коммит (урок 2), push (урок 5), Pull Request и merge (урок 7) — новое только одно: публикация на GitHub Pages. Часть работы в VS Code и терминале (правим файл, коммит, push), часть на сайте GitHub (Pull Request). Самое трудное сегодня — не команды, а аккуратность: ученик создаёт и правит только свой файл `pages/<своя фамилия>.html` и не трогает чужие страницы и `index.html` — иначе ломается общий сайт. На каждом шаге проговаривать, какой файл сейчас открыт.

> Важно весь урок: каждый создаёт и редактирует только свою страницу `pages/<своя фамилия>.html`. Файл `index.html` и чужие страницы трогать нельзя — если их менять, у класса возникнут конфликты и сайт сломается. Имя своего файла (своя фамилия латиницей) преподаватель сообщает каждому заранее, как при подготовке.

#### Смотрим, что уже есть — общий сайт и репозиторий

**Мини-теория:** общий сайт класса уже работает: на нём есть главная страница со списком класса, а личных страниц пока нет. Сначала посмотрим, что уже готово, и где будет наша страница.

**Задание:**

1. Откройте в браузере адрес общего сайта класса (показан на проекторе). Вы увидите главную страницу «Наш класс» со списком учеников.
2. Найдите в списке свою фамилию и нажмите на неё — пока страница не открывается (появляется сообщение, что страница не найдена): вашего файла ещё нет. Это нормально: сегодня вы его создадите, и ссылка заработает.

   ![alt text](image.png)
   
3. Вернитесь назад и откройте в браузере адрес общего репозитория класса (показан на проекторе). Откройте папку `pages` — здесь будут страницы класса. Сейчас в ней лежит только файл-шаблон `temp.html` (из него вы скопируете код); вашего файла пока нет — вы создадите его в практике.

![alt text](image-1.png)

**Ожидаемый результат:** ученик увидел общий сайт со списком класса, понял, что его страницы ещё нет и сегодня он её создаст; нашёл в репозитории папку `pages`, куда добавит свой файл.

#### Скачиваем общий репозиторий к себе (clone)

**Мини-теория:** чтобы удобно править файл в VS Code, скачаем общий репозиторий на компьютер командой `git clone` (как на уроке 6). Разница в том, что теперь у нас есть право записи — мы участники проекта, а не просто гости.

**Задание:**

1. На странице общего репозитория класса нажмите зелёную кнопку **Code**, вкладка **HTTPS**, и нажмите значок копирования рядом с адресом — адрес скопируется.

  ![alt text](image-2.png)

2. Откройте VS Code. В меню выберите **File — Open Folder** и откройте папку `git-clones`, куда мы складывали скачанные проекты на уроке 6 (она в «Документах» или в папке, которую указал тьютор; не на Рабочем столе).
3. Откройте встроенный терминал: меню **Terminal — New Terminal**. Терминал откроется внутри папки `git-clones`.
4. Скачайте репозиторий: напечатайте `git clone ` (с пробелом в конце), затем вставьте скопированный адрес (правой кнопкой мыши — **Paste**, или Ctrl+V) и нажмите Enter.

   ```
   git clone <адрес репозитория>
   ```
5. Внутри `git-clones` появилась папка с именем репозитория (например, `class-site-vtornik`). Слева в панели **Explorer** раскройте её — внутри файлы проекта: `index.html` и папка `pages`. Зайдите в эту папку и в терминале: напечатайте `cd ` и имя папки, нажмите Enter (в начале строки терминала появится имя репозитория).

   ```
   cd class-site-vtornik
   ```

   ![alt text](image-3.png)

**Ожидаемый результат:** репозиторий скачан в папку `git-clones` и виден в VS Code; в панели слева раскрыты `index.html` и папка `pages`, а терминал перешёл внутрь папки проекта.

#### Создаём свою ветку

**Мини-теория:** чтобы предложить изменение через Pull Request и не мешать другим, работаем в отдельной ветке (как на уроке 4). Назовём её по своей фамилии, чтобы не перепутать с чужими.

**Задание:**

1. В терминале VS Code создайте свою ветку и сразу переключитесь в неё (вместо `ivanov` — своя фамилия латиницей):

   ```
   git switch -c page-ivanov
   ```
2. Убедитесь, что переключились: терминал не показал ошибку, а внизу слева в VS Code (синяя полоса) появилось имя вашей ветки.

**Ожидаемый результат:** ученик создал и активировал свою ветку `page-<фамилия>`; понимает, что теперь правки идут в его ветке, а не в общей.

#### Верстаем свою страницу

**Мини-теория:** своей страницы в папке `pages` ещё нет — создадим её сами, а код возьмём из готового шаблона `pages/temp.html`, который уже лежит в репозитории. В шаблоне уже готовая разметка с подсказками-комментариями `<!-- впиши ... -->`; останется заменить прочерки `___` своими словами. Имя своего файла назвал преподаватель (латиницей, маленькими буквами, например `ivanov.html`): оно совпадает со ссылкой на главной странице. Сам шаблон `temp.html` не трогаем — только копируем из него код. Теги в угловых скобках тоже не трогаем, иначе страница сломается.

**Задание:**

1. В панели **Explorer** слева наведите курсор на папку `pages` и нажмите значок создания файла (или правой кнопкой по папке — **New File**). Введите имя своего файла так, как назвал преподаватель, например `ivanov.html` (вместо `ivanov` — своя фамилия латиницей), и нажмите Enter — файл откроется в редакторе.
2. В папке `pages` откройте файл-шаблон `temp.html`, выделите весь его код (Ctrl+A), скопируйте (Ctrl+C). Перейдите на вкладку своего файла и вставьте код (Ctrl+V).
3. Замените прочерки `___` своими словами: впишите своё имя в заголовок, возраст и класс, любимое занятие. Можно добавить ещё один абзац о себе по образцу — скопируйте строку `<p>Я люблю ___.</p>` на новую строку и впишите другой текст между `<p>` и `</p>`.

   ![alt text](image-4.png)

4. Сохраните файл: Ctrl+S (или меню **File — Save**). Если у вкладки файла вместо крестика виден кружок — файл не сохранён, нажмите Ctrl+S ещё раз.

**Ожидаемый результат:** ученик создал в папке `pages` свой файл с назначенным именем, вставил образец и заменил прочерки своим текстом; чужие файлы и `index.html` не тронуты.

#### Коммит и отправка ветки на GitHub

**Мини-теория:** зафиксируем изменение коммитом (урок 2) и отправим свою ветку на GitHub командой `push` (урок 5), чтобы потом предложить её через Pull Request.

**Задание:**

1. В терминале VS Code добавьте изменение к коммиту:

   ```
   git add .
   ```
2. Сделайте коммит с понятным сообщением — укажите в нём имя своего файла (вместо `ivanov.html` — имя своей страницы):

   ```
   git commit -m "Сверстал страницу ivanov.html"
   ```
3. Отправьте свою ветку на GitHub:

   ```
   git push -u origin page-ivanov
   ```

   ![alt text](image-5.png)

> Заметка: при первом `push` GitHub может снова попросить вход через браузер (как на уроке 5). Окно входа иногда прячется за окном VS Code — поищите его на Панели задач Windows внизу экрана.

**Ожидаемый результат:** изменение зафиксировано коммитом, ветка отправлена на GitHub; ученик понимает, что его правка пока лежит в его ветке и ещё не попала в общий сайт.

#### Предлагаем страницу в общий сайт — Pull Request

**Мини-теория:** теперь предложим своё изменение в общий проект через Pull Request (урок 7). На прошлом уроке мы предлагали изменение в чужой форк, а сегодня — внутри одного общего репозитория, из своей ветки в основную ветку `main`. Преподаватель проверит запрос и вольёт его (merge) — и страница появится на сайте.

**Задание:**

1. Откройте в браузере страницу общего репозитория класса. Вверху GitHub покажет жёлтую плашку с кнопкой **Compare & pull request** для вашей только что отправленной ветки — нажмите её. (Если плашки нет — откройте вкладку **Pull requests — New pull request** и выберите **base: main**, **compare: page-ivanov**.)

![alt text](image-6.png)

2. Проверьте направление: **base: main** (куда добавляем) и **compare: page-ivanov** (откуда берём). Должна быть зелёная отметка **Able to merge**.
3. В заголовке коротко напишите, что предлагаете, например: `Сверстал свою страницу`. Нажмите зелёную кнопку **Create pull request**.

![alt text](image-7.png)

4. Поднимите руку — преподаватель проверит ваш Pull Request и вольёт его. После слияния (merge) откройте адрес общего сайта класса, найдите свою фамилию и нажмите — теперь на вашей странице ваш текст.

   ![alt text](image-8.png)

**Действия преподавателя:** по мере поступления Pull Request бегло просматривать вкладку **Files changed** (изменён ли только файл этого ученика, не задет ли `index.html` или чужие страницы) и вливать кнопкой **Merge pull request — Confirm merge**. 

![alt text](image-9.png)

Если ученик случайно изменил чужой файл — не вливать, вернуть на доработку и объяснить. Вливать запросы потоком, не накапливая, чтобы у класса страницы появлялись на сайте по ходу урока.

> Заметка для тьютора про конфликты слияния. При правильной работе merge-конфликтов быть не должно: каждый ученик создаёт и правит только свой новый файл `pages/<фамилия>.html`, а `index.html` собран заранее и никем не редактируется. Разные файлы — разные строки, поэтому Git вливает PR-ы независимо, в любом порядке.
>
> Конфликт (GitHub покажет «Can't automatically merge» и заблокирует кнопку слияния) возникнет только если две влитые ветки изменили одни и те же строки одного файла. Типичные причины на этом уроке:
>
> - двое (или ученик и преподаватель) редактировали `index.html` — например, ученики сами дописывали туда ссылки на свои страницы. Поэтому `index.html` не трогаем.
> - двое создали файл с одинаковым именем (`pages/ivanov.html` у обоих) — заранее назначай каждому уникальное имя файла.
> - кто-то изменил общий шаблон `pages/temp.html` вместо копирования из него.
>
> Если конфликт всё же появился: не вливай такой PR на ходу, отложи его, разберись с учеником после основного потока (проще всего — попросить пересоздать страницу в новом файле с правильным уникальным именем, не трогая `index.html`). Разрешать конфликт руками на уроке с новичками не нужно — это тема за рамками модуля.


**Ожидаемый результат:** Pull Request ученика влит в `main`; личная страница ученика с его текстом опубликована на общем сайте класса; ученик прошёл полный командный цикл от ветки до публикации.

### 4. Самостоятельная работа (35 мин)

**Действия преподавателя:**

Сначала дать ученикам пройти короткий квиз для самопроверки (5 минут) — он закрепляет теорию урока. Разобрать на классе вопросы, где ошиблись многие. Затем основное задание: ученики самостоятельно проходят командный цикл ещё раз — дополняют свою страницу и снова отправляют её через ветку и Pull Request, но теперь вливают сами (они коллабораторы). Это «делаешь сам» после «делаем вместе» в практике. Наблюдать, в какой момент ученики теряются, сразу не подсказывать — навести вопросом (см. «Способы помощи»). Главное, на что смотреть: забирают ли свежую версию (`git pull`) перед новой веткой, создают ли новую ветку, правят ли только свой файл.

#### Квиз для самопроверки

Заметка для вёрстки (mkdocs-programmer): оформить как интерактивный квиз с одним правильным ответом в каждом вопросе. Правильный вариант помечен «(верно)», после каждого вопроса дана строка обратной связи для пояснения после ответа.

**Вопрос 1. Что такое GitHub Pages?**

- А) Программа для рисования сайтов на компьютере
- Б) Бесплатная публикация сайта прямо из репозитория на GitHub (верно)
- В) Команда, которая удаляет страницы сайта
- Г) Платный хостинг, который нужно настраивать вручную

Обратная связь: GitHub Pages бесплатно превращает файлы-страницы из репозитория в настоящий сайт с адресом в интернете.

**Вопрос 2. Какой файл становится главной страницей сайта?**

- А) `main.py`
- Б) `README.md`
- В) `index.html` (верно)
- Г) `style.css`

Обратная связь: `index.html` открывается по основному адресу сайта — это его главная страница.

**Вопрос 3. Почему на этом уроке каждый правит только свой файл в папке `pages`?**

- А) Так быстрее печатать
- Б) Чтобы не возникало конфликтов и общий сайт не сломался (верно)
- В) Это требование браузера
- Г) Чужие файлы вообще нельзя открыть

Обратная связь: если несколько человек правят один и тот же файл, возникают конфликты; правя только свой файл, мы их избегаем.

**Вопрос 4. Когда твоя страница появится на общем сайте?**

- А) Сразу, как только ты сохранил файл в VS Code
- Б) После того как Pull Request вольют (merge) в ветку `main` (верно)
- В) Никогда, сайт обновляет только преподаватель вручную
- Г) После перезагрузки компьютера

Обратная связь: страница публикуется, когда изменение из ветки попадает в `main` через merge — тогда GitHub Pages обновляет сайт.

#### Задание

Дополни свою страницу и снова проведи изменение через полный цикл — теперь самостоятельно, от ветки до публикации.

1. В терминале VS Code забери свежую версию общего проекта (там уже есть твоя страница с практики):

   ```
   git switch main
   git pull
   ```
2. Создай новую ветку под доработку (вместо `ivanov` — своя фамилия):

   ```
   git switch -c page-ivanov-extra
   ```
3. Открой свой файл `pages/<своя фамилия>.html` и добавь о себе что-то ещё: список любимых книг, игр или предметов. Подсказка: список делают так —

   ```html
   <p>Мои любимые предметы:</p>
   <ul>
     <li>математика</li>
     <li>история</li>
   </ul>
   ```
   Сохрани файл (Ctrl+S). Правь только свой файл.
4. Закоммить и отправь ветку:

   ```
   git add .
   git commit -m "Добавил список любимого на свою страницу"
   git push -u origin page-ivanov-extra
   ```
5. На GitHub открой свой репозиторий, нажми **Compare & pull request**, проверь направление (**base: main**, **compare: page-ivanov-extra**) и нажми **Create pull request**.
6. Сделай себе короткий обзор: открой вкладку **Files changed**, перечитай своё изменение — изменён ли только твой файл, нет ли опечаток в тегах. Если всё хорошо, вернись на вкладку **Conversation** и нажми **Merge pull request**, затем **Confirm merge**.
7. Открой адрес общего сайта класса, найди свою страницу и обнови её (F5) — на ней появилось добавленное.
8. Покажи преподавателю: влитый (Merged) Pull Request и обновлённую страницу на сайте.

> Заметка: иногда сайт обновляется не мгновенно — GitHub Pages нужна минута. Если изменений ещё не видно, подожди немного и обнови страницу (F5).

#### Критерии оценки

| Результат | Оценка |
|-----------|--------|
| Самостоятельно прошёл весь цикл: забрал свежую версию, создал новую ветку, дополнил только свою страницу, отправил ветку, открыл Pull Request, сам сделал merge; изменение видно на сайте | Отлично |
| Выполнил всё, но с одной-двумя мелкими ошибками или с одной подсказкой (например, забыл `git pull` или сделал правку без новой ветки) | Хорошо |

## Методические заметки преподавателя

### Возможные сложности

- Не принято приглашение в коллабораторы. Без права записи ученик не сможет отправить ветку (`push` выдаст ошибку доступа). Проверить на оргмоменте; кто не принял — пусть откроет уведомления на GitHub или письмо на почте и нажмёт **Accept invitation**.
- Правят чужой файл, `index.html` или сам шаблон `temp.html`. Это главный риск варианта с общим репозиторием: чужие правки рушат сайт и создают конфликты. Шаблон `temp.html` — только для копирования кода, его не меняют. Постоянно проговаривать: открыт ли свой файл (проверь фамилию на вкладке). При обзоре Pull Request преподаватель отклоняет запросы, где задет не свой файл.
- Кириллица или пробел в имени файла. Адреса сайта плохо работают с кириллицей и пробелами — поэтому имя своего файла каждому преподаватель называет заранее, латиницей и маленькими буквами. Ученик создаёт файл ровно с этим именем, не придумывает своё и не использует кириллицу или пробелы; иначе ссылка из `index.html` не сработает.
- Ошибка доступа при `push`. Чаще всего — не принято приглашение или Git снова просит вход через браузер. Окно входа прячется за VS Code — искать на Панели задач Windows. Проверить, что ученик в общем репозитории класса, а не в чужом.
- Забывают создать ветку и правят прямо в `main`. Тогда нечего предлагать через Pull Request, а правка идёт в общий проект напрямую. Напоминать порядок: сначала `git switch -c`, потом правка.
- Забывают `git pull` перед самостоятельной работой. Без свежей версии новая ветка отойдёт от старого состояния. Напоминать: перед новой веткой — `git switch main` и `git pull`.
- Ломают разметку HTML — стёрли угловую скобку или тег. Страница начинает выглядеть неправильно. Объяснить: менять только текст между тегами, прочерки `___`; теги в угловых скобках не трогать. На обзоре Pull Request это видно на вкладке **Files changed**.
- Сайт не обновился сразу. GitHub Pages обновляет сайт с задержкой до минуты. Подождать и обновить страницу (F5), не паниковать и не отправлять изменение повторно.
- Очередь на merge у преподавателя. Если запросы вливать не сразу, страницы не появляются и дети ждут. Вливать потоком; в помощь привлечь сильных учеников, кто уже закончил, — пусть помогают соседям, пока преподаватель вливает.

### Способы помощи

Подсказки давать по нарастанию, не выдавая ответ сразу:

- При ошибке доступа на push: «Ты принял приглашение в общий репозиторий? Открой на GitHub уведомления — нет ли там Accept invitation. И проверь, что отправляешь ветку именно в общий репозиторий класса».
- Если открыт не тот файл: «Посмотри на вкладку вверху — твоя ли там фамилия? Менять можно только свой файл в папке `pages`».
- Если нет кнопки для Pull Request: «Ты отправил ветку командой `git push`? И создал ли ты ветку через `git switch -c` перед правкой? Кнопка появляется, только когда есть отправленная ветка с изменением».
- Если страница на сайте выглядит сломанной: «Открой вкладку Files changed в своём Pull Request. Не стёрлась ли где угловая скобка или тег? Менять нужно только текст, а не теги».
- Если в самостоятельной работе изменения «потерялись»: «Ты забрал свежую версию через `git pull` и создал новую ветку? Старая ветка с практики уже влита — для новой правки нужна новая ветка».
- Если сайт не обновился: «Подожди минуту и обнови страницу клавишей F5 — Pages обновляет сайт не мгновенно».
