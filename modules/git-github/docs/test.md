# Контрольная работа — От Git до GitHub

## Общая информация

| Параметр          | Значение                                  |
| ----------------- | ----------------------------------------- |
| Модуль            | От Git до GitHub                           |
| Тема              | Полный цикл работы: локальный Git → GitHub |
| Возраст учащихся  | 12–14 лет                                  |
| Продолжительность | 120 мин (90 мин — работа, 30 мин — проверка) |
| Форма             | Практическая, на компьютере               |
| Максимальный балл | 100                                        |

Контрольная работа полностью практическая, письменных вопросов нет. Ученик
проходит весь путь модуля на новом мини-проекте `git-control`: от первого
коммита до командной работы через Pull Request на GitHub. Каждый шаг оставляет
проверяемый след в репозитории. Преподаватель проверяет работу по ссылке на
репозиторий, сверяясь с чек-листом (см. «Методические заметки»).

Задания для учеников оформлены отдельным файлом «Контрольная работа — От Git
до GitHub.pdf» (см. «Материалы урока»): тьютор распечатывает его и раздаёт
ученикам или выводит на проектор.

---

## Цель контрольной работы

Проверить, что ученик самостоятельно, без подсказок, может:

- создать локальный репозиторий и фиксировать изменения коммитами;
- работать с ветками: создать, влить в основную, удалить;
- безопасно отменить ошибочный коммит командой `revert`;
- опубликовать проект на GitHub (`remote add`, `push`);
- вести командную работу: создать Issue и провести изменение через Pull Request.

## Задания

Все команды Git набирай в терминале VS Code (меню **Terminal — New Terminal**).
Код из заданий копируй в файлы точно как написано.

### Задание 1 — Локальный репозиторий и первые коммиты

**Баллов:** 15

**Инструкция для учащегося:**

1. В папке, где ты хранишь свои проекты (например, «Документы» или папку, которую укажет тьютор, — не на Рабочем столе), создай папку `git-control` (английскими буквами, без пробелов) и открой её в VS Code (**File — Open Folder**).
2. Сделай из этой папки репозиторий Git.
3. Создай в папке файл `program.py` и впиши в него код ниже. Зафиксируй файл первым коммитом с понятным сообщением.
4. Допиши в `program.py` вторую функцию (код ниже) и сделай отдельный второй коммит.

**Содержание задания:**

Первый коммит — файл `program.py`:

```python
def add(a, b):
    return a + b
```

Второй коммит — допиши ниже:

```python
def subtract(a, b):
    return a - b
```

**Проверяемый артефакт / критерии проверки:**

- В `git log` не меньше двух коммитов с осмысленными сообщениями (5 б).
- Каждая функция добавлена своим отдельным коммитом, а не обе сразу (5 б).
- В файле `program.py` есть обе функции `add` и `subtract` (5 б).

---

### Задание 2 — Ветка и слияние

**Баллов:** 15

**Инструкция для учащегося:**

1. Создай новую ветку `feature-multiply` и сразу перейди в неё.
2. Допиши в `program.py` функцию `multiply` (код ниже) и сделай в ветке коммит.
3. Вернись в основную ветку и убедись, что функции `multiply` там пока нет.
4. Влей ветку `feature-multiply` в основную.
5. Удали ветку `feature-multiply`.

**Содержание задания:**

```python
def multiply(a, b):
    return a * b
```

**Проверяемый артефакт / критерии проверки:**

- Функция `multiply` появилась в основной ветке после слияния (6 б).
- В истории видно, что коммит пришёл из ветки `feature-multiply` (5 б).
- Ветка `feature-multiply` удалена (`git branch` её больше не показывает) (4 б).

---

### Задание 3 — Отмена ошибочного коммита

**Баллов:** 15

**Инструкция для учащегося:**

1. Допиши в `program.py` «ошибочную» функцию `divide` (код ниже) и сделай коммит — представь, что ошибку заметили уже после фиксации.
2. Безопасно отмени именно этот коммит, **не стирая** историю (так, чтобы и ошибочный коммит, и его отмена остались видны).

**Содержание задания:**

```python
def divide(a, b):
    return a / 0
```

**Проверяемый артефакт / критерии проверки:**

- В истории есть и ошибочный коммит, и отдельный коммит-отмена (`revert`) (9 б).
- В итоговом файле `program.py` функции `divide` нет (6 б).

> Если ученик стёр коммит так, что в истории не осталось следа (например, удалил строку и переписал историю) — задание не засчитывается: проверялась именно безопасная отмена через `revert`.

---

### Задание 4 — Публикация на GitHub

**Баллов:** 25

**Инструкция для учащегося:**

1. Зайди на свой аккаунт GitHub и создай **пустой** репозиторий с именем `git-control` (без README, без лицензии).
2. Привяжи к нему свой локальный репозиторий.
3. Отправь на GitHub всю историю проекта.
4. Обнови страницу репозитория на сайте и убедись, что файлы и все коммиты на месте.

**Проверяемый артефакт / критерии проверки:**

- Репозиторий `git-control` доступен по ссылке (6 б).
- На GitHub приехал файл `program.py` (6 б).
- На GitHub видна вся история коммитов из заданий 1–3, включая `revert` (13 б).

---

### Задание 5 — Issue

**Баллов:** 10

**Инструкция для учащегося:**

На странице репозитория `git-control` открой вкладку **Issues** и создай новую
Issue: предложи понятную идею доработки калькулятора. Заголовок и описание должны
ясно объяснять, что нужно сделать (например, «Добавить функцию возведения в
степень `power(a, b)`»).

**Проверяемый артефакт / критерии проверки:**

- В репозитории создана Issue (4 б).
- У Issue понятный заголовок (3 б).
- В описании ясно сказано, что нужно сделать — не пусто и не «тест» (3 б).

---

### Задание 6 — Pull Request

**Баллов:** 20

**Инструкция для учащегося:**

1. На компьютере создай новую ветку `feature-readme` и перейди в неё.
2. Создай в папке файл `README.md` и напиши в нём одну-две строки — что это за проект.
3. Сделай коммит и отправь ветку `feature-readme` на GitHub.
4. На сайте GitHub открой **Pull Request** из ветки `feature-readme` в основную ветку.
5. Влей этот Pull Request (кнопка **Merge pull request**).

**Проверяемый артефакт / критерии проверки:**

- На GitHub есть Pull Request из ветки `feature-readme` (6 б).
- Pull Request влит (статус **Merged**) (7 б).
- Файл `README.md` появился в основной ветке репозитория (7 б).

## Методические заметки

### Чек-лист проверки (на каждого ученика)

Открыть ссылку ученика на репозиторий и пройти по пунктам. Один пункт — один артефакт, проверка занимает 15–25 секунд.

| № | Что проверяем (где смотреть)                                   | Балл |
| - | -------------------------------------------------------------- | ---- |
| 1 | ≥2 коммита с осмысленными сообщениями, обе функции (вкладка Commits) | 15 |
| 2 | `multiply` в основной ветке, виден след ветки, ветка удалена   | 15   |
| 3 | В истории есть `revert`-коммит, функции `divide` в файле нет    | 15   |
| 4 | Репозиторий онлайн, `program.py` и вся история на месте         | 25   |
| 5 | Создана понятная Issue                                          | 10   |
| 6 | Pull Request из `feature-readme` влит, `README.md` в main       | 20   |
|   | **Итого**                                                      | 100  |

Проверка 12 работ по этому чек-листу укладывается в 25–30 минут: все артефакты
видны на странице репозитория `git-control`. Открывать его можно со своего
компьютера (аккаунты учеников известны с урока 5) или подойдя к ученику, который
поднял руку, — на балл это не влияет.

### Типичные ошибки учащихся

- Обе функции в задании 1 фиксируют одним коммитом — теряют баллы за «отдельный коммит».
- В задании 3 пытаются «удалить» ошибку правкой файла без `revert` — это не отмена коммита.
- Забывают сделать `push` после локальных коммитов — на GitHub приезжает не вся история (частая причина потери баллов в заданиях 4 и 6).
- В задании 6 открывают Pull Request, но не нажимают **Merge** — статус остаётся открытым.

### На что обратить внимание при проверке

- Главный показатель навыка — задание 4: если на GitHub приехала **вся** история коммитов (включая ветку и `revert`), значит локальная часть выполнена верно.
- Имя репозитория должно быть ровно `git-control` — так быстрее находить и сверять работы.
- Проблемы с авторизацией при первом `push` (запрос логина/токена GitHub) — это техническая заминка, а не ошибка навыка: можно помочь с входом, на балл это не влияет.
- Если ученик быстро закончил — предложить дополнительно: добавить через Pull Request функцию из своей же Issue (задание 5) и закрыть Issue ссылкой на этот PR.

### После контрольной работы

- Ссылки на репозитории собраны, работы проверены по чек-листу.
- Результаты и оценки зафиксированы.
- Разбор типичных ошибок (отдельный коммит, `revert`, забытый `push`, merge PR) проведён на следующем занятии.
- Учащиеся получили обратную связь.
